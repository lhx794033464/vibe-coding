## 二、CRITICAL 级别问题

### C-1：S3 凭据硬编码为空字符串（多处）

**文件**：
- `src/app/api/acceptance-doc/upload/route.ts` (L9-10)
- `src/app/api/acceptance-doc/view/route.ts` (L8-9)
- `src/app/api/dismissal-applications/route.ts` (L8-9)
- `src/app/api/dismissal-applications/kbc-screenshot/route.ts` (L7-8)
- `src/app/api/process-applications/route.ts` (L8-9)

**问题**：S3Storage 初始化时 `accessKey` 和 `secretKey` 硬编码为空字符串，未使用环境变量。

```typescript
const storage = new S3Storage({
  endpointUrl: process.env.COZE_BUCKET_ENDPOINT_URL,
  accessKey: '',  // ❌ 应为 process.env.COZE_BUCKET_ACCESS_KEY
  secretKey: '',  // ❌ 应为 process.env.COZE_BUCKET_SECRET_KEY
  bucketName: process.env.COZE_BUCKET_NAME,
  region: 'cn-beijing',
});
```

**风险**：S3 功能完全不可用；若 SDK 有默认凭证链回退行为，可能导致未授权访问。  
**修复建议**：
```typescript
const storage = new S3Storage({
  endpointUrl: process.env.COZE_BUCKET_ENDPOINT_URL,
  accessKey: process.env.COZE_BUCKET_ACCESS_KEY || '',
  secretKey: process.env.COZE_BUCKET_SECRET_KEY || '',
  bucketName: process.env.COZE_BUCKET_NAME,
  region: 'cn-beijing',
});
```

### C-2：AuthContext 中 `atob()` 将 JWT 作为 Base64 解码，Session 恢复功能失效

**文件**：`src/contexts/AuthContext.tsx` (L82-98)

**问题**：Cookie 恢复会话时使用 `atob(cookieToken)` 解码 JWT Token。JWT 由三段 Base64URL 编码组成，直接用 `atob()` 解码整个 JWT 字符串会产生乱码或异常，导致 Session 恢复失败。

```typescript
// ❌ 错误：JWT 不是简单的 Base64 编码
const cookieToken = document.cookie
  .split('; ')
  .find(row => row.startsWith('app_auth='))
  ?.split('=')[1];
if (cookieToken) {
  const decoded = JSON.parse(atob(cookieToken)); // JWT 不可以这样解析
```

**风险**：用户刷新页面后无法恢复登录状态，每次都需要重新登录。  
**修复建议**：移除 `atob` 解码逻辑，JWT 只需直接作为 token 使用，不需要客户端解码。

---

### C-3：JWT_SECRET 使用硬编码回退值

**文件**：`src/services/dbService.ts` (L10)

**问题**：
```typescript
const JWT_SECRET = process.env.JWT_SECRET || 'kingdee-xingchen-delivery-platform-secret-key-change-in-production';
```

**风险**：如果环境变量未设置，所有 JWT 将使用公开的默认密钥签名，攻击者可以伪造任意 token。  
**修复建议**：启动时检查 `JWT_SECRET` 是否已设置，未设置则抛出异常阻止启动。

### C-4：Migrate API 允许未认证用户执行数据迁移

**文件**：`src/app/api/migrate/route.ts` (L19, L53)

**问题**：使用可选链 `userInfo?.id` 处理用户信息，当 `getCurrentUserInfo` 返回 null 时（未认证），迁移操作仍可执行。

```typescript
const userInfo = await getCurrentUserInfo(request);
// ...
user_id: userInfo?.id || null,  // ❌ 未认证时 userInfo 为 null，.id 为 undefined，仍可迁移
```

**风险**：任何人无需登录即可将数据迁移到系统中，可能注入恶意数据。  
**修复建议**：在迁移前强制验证用户认证状态。

### C-5：Search API 无认证检查

**文件**：`src/app/api/search/route.ts`

**问题**：搜索 API 完全没有认证检查，任何人都可以调用。

**风险**：搜索 API 调用了外部服务（Coze SearchClient），未认证调用可能导致：
1. 外部 API 配额耗尽
2. 敏感搜索词泄露
3. 服务器资源被滥用

**修复建议**：添加 `getCurrentUserInfo(request)` 认证检查。

## 三、MAJOR 级别问题

### M-1：supabaseUsersService.ts 中 `generateId()` 仍使用 `Math.random()`

**文件**：`src/services/supabaseUsersService.ts` (L32-33)

**问题**：
```typescript
function generateId(): string {
  return crypto.randomUUID ? crypto.randomUUID() : Math.random().toString(36).substring(2);
}
```

**风险**：`Math.random()` 不具备加密安全性，ID 可被预测。  
**修复建议**：移除 `Math.random()` 回退，在 Node.js 环境中 `crypto.randomUUID()` 始终可用。

---

### M-2：Login/Register/Chat 等敏感 API 无速率限制

**文件**：
- `src/app/api/auth/login/route.ts`
- `src/app/api/auth/register/route.ts`
- `src/app/api/chat/route.ts`
- `src/app/api/chat/qa/route.ts`

**问题**：登录、注册、AI 对话等 API 没有任何速率限制。

**风险**：暴力破解密码、批量注册账号、消耗 AI API 配额。  
**修复建议**：引入速率限制中间件（如 `next-rate-limit` 或自定义 Redis 计数器）。

---

### M-3：Login API 每次请求都调用 `ensureAdminUser()`

**文件**：`src/app/api/auth/login/route.ts` (L11)

**问题**：每次登录请求都执行 `ensureAdminUser()`，该函数会查询数据库检查管理员是否存在，可能还会尝试写入。

**风险**：高并发登录场景下产生不必要的数据库查询，影响性能。  
**修复建议**：将 `ensureAdminUser()` 移至应用启动时执行一次（如 `instrumentation.ts` 或中间件初始化）。

---

### M-4：KBC 截图签名 URL 无权限隔离

**文件**：`src/app/api/dismissal-applications/kbc-screenshot/route.ts` (L22-31)

**问题**：任何已登录用户只要知道文件 `key`，就能获取签名 URL。

```typescript
const key = searchParams.get('key'); // 任何 key 值均可通过
const url = await storage.generatePresignedUrl({ key, expireTime: 3600 });
```

**风险**：IDOR（不安全的直接对象引用），用户可访问其他人的截图。  
**修复建议**：验证当前用户是否为申请人或管理员，再返回签名 URL。

### M-5：Process Applications 文件名使用 `Math.random()`

**文件**：`src/app/api/process-applications/route.ts` (L83)

```typescript
const randomStr = Math.random().toString(36).substring(2, 10);
const fileName = `kbc-screenshots/${timestamp}_${randomStr}.${ext}`;
```

**风险**：文件名可被预测，`Math.random()` 不是安全随机源。  
**修复建议**：使用 `crypto.randomUUID()` 生成不可预测的文件名。

### M-6：Commissions POST 中提成计算逻辑与 `calculateCommission` 函数不一致

**文件**：`src/app/api/commissions/route.ts` (L218-226 vs L7-51)

**问题**：`calculateCommission()` 函数使用 `COMMISSION_CONFIG` 配置计算提成，但 POST 处理函数内联了不同的计算逻辑（`implementationFee > 500` 阈值，硬编码费率 0.3/0.4/0.5）。

**风险**：前端显示的提成金额与后端实际保存的不一致，可能导致财务数据错误。  
**修复建议**：POST 处理函数应复用 `calculateCommission()` 函数，保持逻辑一致性。

### M-7：Tencent Docs Config API 非管理员可删除全局配置

**文件**：`src/app/api/tencent-docs/config/route.ts` (L195-202)

**问题**：DELETE 端点只检查了用户是否登录，任何用户都能删除腾讯文档全局 Token 配置。

**风险**：普通用户误操作删除系统级配置，导致同步功能中断。  
**修复建议**：限制 DELETE 操作仅管理员可用。

---

### M-8：`execFile` 调用 Python 脚本存在命令注入风险

**文件**：`src/app/api/chat/qa/route.ts` (L17-38)

**问题**：用户输入的 `question`、`token`、`productId` 等参数直接传递给 `execFile`。虽然 `execFile` 比 `exec` 安全（不通过 shell 执行），但 `--question` 和 `--save-token` 的参数值来自用户输入，若 Python 脚本内部拼接命令则仍有风险。

**修复建议**：
1. 对 `question` 参数做长度限制（如最多 500 字符）
2. 对 `token` 参数做格式校验
3. 确认 Python 脚本内部使用 `subprocess.run` 而非 `os.system` 或字符串拼接

---

### M-9：Dashboard 非管理员获取全部客户后在应用层过滤

**文件**：`src/app/api/dashboard/route.ts` (L63-71)

**问题**：非管理员获取全部客户数据后，在应用层按 `delivery_consultant` 过滤。

```typescript
const allCustomers = await dbGetCustomers({ isAdmin: true });
customers = allCustomers.filter((c: any) => c.delivery_consultant === username);
```

**风险**：
1. 查询了不需要的数据，浪费数据库资源
2. 按 `delivery_consultant`（用户名）匹配而非 `user_id`，如果用户名变更则数据隔离失效
3. 短暂的时间内所有客户数据暴露在内存中

**修复建议**：在数据库查询层即传入 `userId` 过滤，避免全量加载。

## 四、MINOR 级别问题

### m-1：大量使用 `(customer as any)` 类型断言

**文件**：`acceptance-doc/route.ts`、`dashboard/route.ts`、`commissions/route.ts` 等

**问题**：代码中大量使用 `as any` 类型断言绕过类型检查。

**风险**：失去 TypeScript 类型安全保护，属性名拼写错误不会被编译器捕获。  
**修复建议**：定义完整的 Customer 类型接口，替换 `as any`。

---

### m-2：dbService.ts 中 `verifyPassword` 的 Base64 兼容逻辑

**文件**：`src/services/dbService.ts` (L34-42)

**问题**：为兼容旧密码，使用 `Buffer.from(hash, 'base64').toString('utf-8')` 解码后与明文比较。

```typescript
// Legacy Base64 passwords
const decoded = Buffer.from(hash, 'base64').toString('utf-8');
if (decoded === password) {
  // Auto-upgrade to bcrypt
```

**风险**：旧的 Base64 密码存储方式极不安全，任何数据库泄露可直接还原明文。  
**修复建议**：设定迁移截止日期，强制所有仍使用 Base64 密码的用户重置密码。

---

### m-3：Holidays API 无认证检查

**文件**：`src/app/api/holidays/route.ts`

**问题**：节假日查询 API 无需登录即可调用，虽然数据不敏感，但外部 API（timor.tech）调用可被滥用。

**修复建议**：添加简单认证检查，或使用内存缓存减少外部 API 调用。

---

### m-4：内存缓存无上限控制

**文件**：`src/app/api/holidays/route.ts` (L4)

```typescript
const cache: Record<number, { data: HolidayData; timestamp: number }> = {};
```

**问题**：年份参数虽然限制了 2020-2030，但缓存对象没有大小限制，理论上可无限增长。

**修复建议**：对于年份有限的数据这不太可能出问题，但建议添加 LRU 淘汰策略或显式上限。

---

### m-5：Schedule POST 不验证用户身份

**文件**：`src/app/api/schedule/route.ts` (L101-118)

**问题**：创建日程时，如果 `getCurrentUserInfo` 返回 null，`user_id` 会被设为 `null`，日程将成为"无主"数据。

```typescript
const userInfo = await getCurrentUserInfo(request);
// 无 null 检查
user_id: userInfo?.id || null,
```

**修复建议**：添加 `if (!userInfo) return 401` 检查。

### m-6：Implementation Logs POST 同样不验证用户身份

**文件**：`src/app/api/implementation-logs/route.ts` (L42-51)

**问题**：与 Schedule POST 相同，`userInfo?.id` 可能为 undefined，`user_id` 会设为 null。

**修复建议**：添加认证检查。

## 五、安全架构总结

### 已做好的方面 ✅

1. **JWT 认证体系**：使用 bcrypt + JWT，24h token 过期
2. **数据隔离**：大部分 API 实现了基于 `user_id` 的数据隔离
3. **二次验证**：`serverAuth.ts` 在 JWT 验证后还会查询数据库确认用户 `is_active`
4. **安全头配置**：`next.config.ts` 正确配置了 X-Content-Type-Options、X-Frame-Options 等
5. **文件上传校验**：验收文档上传有类型和大小校验
6. **XSS 防护**：客户搜索已对 `%` 和 `_` 通配符做转义
7. **Content-Disposition 编码**：文件下载正确使用 `filename*=UTF-8''` + `encodeURIComponent`
8. **管理员保护**：删除最后一个管理员有保护
9. **Debug API 已禁用**：`/api/debug/supabase` 返回 404

### 需要改进的方面 ❌

1. **S3 凭据管理**：5 处 S3Storage 使用空字符串而非环境变量
2. **速率限制**：所有敏感 API 均未实现
3. **IDOR 风险**：KBC 截图签名 URL 无资源归属验证
4. **JWT 回退密钥**：硬编码回退值暴露于源码
5. **Session 恢复**：`atob()` 解码 JWT 导致功能失效
6. **计算逻辑不一致**：提成计算存在两套不同逻辑

你在嵌入式 draw.io 中调用 DeepSeek API 生成流程图时，发现生成质量不如官网对话、速度也更慢。这通常不是 DeepSeek 本身能力不足，而是 **API 调用方式、提示词设计或参数配置** 未能充分释放其潜力。下面我从几个核心维度分析原因，并给出优化建议。

---

## 🔍 一、提示词工程（最关键）

**官网对话** 通常有精心设计的系统提示，引导模型输出结构化、高质量的内容。而你在 API 中可能只用了简单的系统提示，导致模型理解偏差或输出不完整。

**优化方法**：
1. **明确输出格式**：要求 DeepSeek 输出 **纯 XML**，不加任何额外文字、Markdown 标记。
2. **提供示例**：在提示词中嵌入一个简单的 draw.io XML 示例，让模型参考结构。
3. **强调布局要求**：明确要求“自上而下垂直布局，连线正交（无斜线）”。
4. **使用系统角色**：在 `messages` 中设置 `role: "system"` 来固定行为，与用户消息区分。

**改进后的系统提示示例**：
```text
你是一个专业的流程图生成助手。用户会描述一个流程，你需要生成 draw.io 可加载的 mxGraphModel XML 代码。要求：
- 只输出 XML，不包含任何解释、Markdown 或代码块标记。
- 布局为自上而下垂直排列，所有连线使用正交路由（无斜线）。
- 节点名称清晰，流程逻辑正确。
- 示例格式（仅供参考）：
<mxGraphModel>
  <root>
    <mxCell id="0"/>
    <mxCell id="1" parent="0"/>
    <mxCell id="2" value="开始" style="rounded=1;whiteSpace=wrap;html=1;" vertex="1" parent="1">
      <mxGeometry x="120" y="40" width="80" height="40" as="geometry"/>
    </mxCell>
    ...
  </root>
</mxGraphModel>
现在请根据以下描述生成 XML：
```

---

## ⚙️ 二、API 参数设置

官网对话可能使用了更优的生成参数，而你的调用可能使用了默认值或不当值。

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `temperature` | **0.1 ~ 0.3** | 越低输出越确定，适合结构化生成。官网对话可能用 0.2 左右。 |
| `top_p` | **0.95** | 保持默认或与 temperature 配合。 |
| `max_tokens` | **4000 ~ 8000** | 确保足够生成完整 XML，避免截断。流程图可能产生大量 token。 |
| `model` | `deepseek-chat` | 确保使用最新版本（如 `deepseek-chat`，而非旧版）。官网通常用最新模型。 |

**检查代码**：
```javascript
const response = await axios.post(DEEPSEEK_URL, {
  model: 'deepseek-chat',
  messages: [...],
  temperature: 0.2,
  top_p: 0.95,
  max_tokens: 6000
});
```

---

## ⏱️ 三、响应速度与流式输出

官网对话采用 **流式输出**（streaming），用户能逐步看到内容，感觉更快；而你的 API 调用是 **一次性等待全部生成**，会感觉更慢。

**优化建议**：
- 如果环境支持，可启用 `stream: true`，并在前端逐步接收内容（但需要额外处理 XML 片段组装）。对于流程图生成，流式意义不大，可以接受完整返回。
- 检查 API 调用网络延迟：国内访问 DeepSeek API 通常很快，但若通过代理或云函数可能增加延迟。可尝试直接在前端调用（但 API Key 会暴露，不安全）。

---

## 📦 四、模型版本与上下文

- **确认模型版本**：DeepSeek 有多个版本（V2、V3、R1 等），`deepseek-chat` 目前指向最新稳定版。官网可能使用了更强版本（如 DeepSeek-V3.2）。可尝试指定具体版本（如有）。
- **上下文长度**：流程图 XML 可能很长，确保 `max_tokens` 足够大。若生成的 XML 被截断，draw.io 加载会失败。建议设置 `max_tokens` 为 6000 或更高。

---

## 🧹 五、输出后处理

你可能对 API 返回的内容做了清理（如去除 markdown 代码块），但如果模型输出了不完整或格式错误的 XML，会导致加载失败。需要增加 **XML 合法性校验**，若无效可重试或提示用户。

**建议**：
- 先打印原始返回内容，检查是否包含 `<mxGraphModel>` 根元素。
- 使用正则或 DOMParser 验证 XML 格式。
- 如果模型输出了额外的解释文本，尝试提取其中 `<mxGraphModel>...</mxGraphModel>` 部分。

---

## 🧪 六、与官网对话对比测试

为确认问题，你可以进行 **相同输入的对比测试**：

1. 在官网对话中输入你的描述，查看生成的 XML 质量。
2. 用 API 调用相同的描述，比较输出结果。
3. 如果官网输出明显更好，说明你的 API 调用参数或提示词需要调整。如果官网输出同样不佳，可能是模型本身对流程图 XML 生成不够擅长，可考虑改用其他模型（如 GPT-4 或 Claude）或使用专门的图表生成工具。

---

## 🚀 七、替代方案：使用 Mermaid 中转

如果 DeepSeek 直接生成 draw.io XML 困难，可以考虑让它生成 **Mermaid 代码**，然后在前端利用库将 Mermaid 转换为 draw.io XML（例如使用 `mermaid` 库的 `render` 功能，再提取 SVG 手动构造 XML，但较为复杂）。更简单的是让 draw.io 直接支持 Mermaid 加载？但 draw.io 嵌入模式原生不支持 Mermaid 语法加载（需借助插件）。因此直接生成 XML 仍是首选。

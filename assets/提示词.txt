---

## 📌 需求说明
在现有嵌入式 draw.io 编辑器的基础上，当用户点击“生成流程图”后，希望能在编辑器中**实时看到节点和连线逐步出现**，而不是等待全部生成完成后一次性加载。

---

## 🧠 技术难点分析
- draw.io 的 `postMessage` 接口只提供 `load` 动作，需要一次性传入完整的 XML，无法增量添加节点。
- 要实现“实时预览”，必须在后端流式生成 XML 片段，前端逐步接收并**重新加载完整 XML**（会闪烁且丢失视图状态），或者利用 draw.io 内部的 API 动态操作模型（复杂度高，且嵌入模式下未公开）。

因此，**完美实时预览较难实现**。以下提供两种折中方案，可根据体验要求选择。

---

## 🛠️ 方案一：模拟实时预览（推荐）
**原理**：在生成过程中，前端不直接更新 draw.io，而是显示一个进度条或动画，让用户感知到生成过程。生成完成后，一次性加载最终 XML。用户依然获得“等待→呈现”的流畅体验，没有闪烁。

**实现步骤**：
1. 前端点击“生成流程图”，显示加载动画或进度提示。
2. 调用后端接口，后端一次性生成完整 XML 并返回。
3. 前端收到完整 XML 后，通过 `postMessage` 加载到 draw.io，同时隐藏加载动画。

**优点**：实现简单，体验稳定，无闪烁。  
**缺点**：不是真正的实时预览，但用户感知不明显。

---

## ⚙️ 方案二：增量式伪实时预览（高级）
**原理**：将流程拆分为多个阶段，每次生成一部分节点/连线，前端每次收到一个阶段的数据后，将当前所有已生成的节点合并为完整 XML 重新加载。由于 XML 结构在逐步完善，编辑器会刷新，但用户能看到内容逐渐增多。

**实现步骤**：
1. 后端采用流式输出，每次发送一个 JSON 块，包含当前阶段的所有节点和连线（如 `{ type: 'node', data: {...} }`）。
2. 前端维护一个累积的 `nodes` 和 `edges` 数组，每收到一块，重新构建完整 XML，并调用 `load` 更新 draw.io。
3. 为了减少闪烁，可在更新前后保存当前视图位置和缩放比例（draw.io 支持通过 `postMessage` 获取/设置视图状态）。

**优点**：实现了“逐步出现”的视觉效果。  
**缺点**：每次重新加载都会重置编辑器视图（除非手动恢复），且实现复杂；如果生成块较多，频繁刷新可能导致体验下降。
